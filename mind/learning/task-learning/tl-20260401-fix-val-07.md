## Candidate Knowledge

### [KDC-001] Redlock Algorithm — Distributed Mutual Exclusion via Independent Redis Masters

**Source Type**: grounded-external

- **Acquisition Event**: ACQ-001
- **Source Anchor**: `tasks/20260401-fix-val-07/acquire/raw-sources/src-001-redlock-specification.md`
- **Verification Status**: passed
- **Original Excerpt**: > "Purpose: implement a distributed, fault-tolerant mutual exclusion lock using multiple independent Redis nodes. Basic idea: acquire the same logical lock in a majority of independent Redis servers, each for a short TTL. If you manage to lock a majority and the total time to acquire the locks is less than the TTL (adjusted for clock drift), you consider the lock acquired."
- **Derived Conclusion**: Redlock achieves distributed mutual exclusion by requiring a client to acquire locks on a majority (>N/2) of N independent Redis masters within a validity window calculated as TTL minus elapsed time minus clock drift. The algorithm does not use consensus — instead it relies on independent, uncoordinated masters and a quorum-based success decision. Safety depends on bounded clock drift and the unique token mechanism for atomic release.

### [KDC-002] ZooKeeper Lock Recipe — Ephemeral Sequential Znodes with Watch-Predecessor Pattern

**Source Type**: grounded-external

- **Acquisition Event**: ACQ-002
- **Source Anchor**: `tasks/20260401-fix-val-07/acquire/raw-sources/src-004-zookeeper-lock-recipes.md`
- **Verification Status**: passed
- **Original Excerpt**: > "Fully distributed locks that are globally synchronous, meaning at any snapshot in time no two clients think they hold the same lock. These can be implemented using ZooKeeper."
- **Derived Conclusion**: ZooKeeper implements distributed locks as recipes built from primitive operations: clients create ephemeral sequential znodes under a lock path, the client with the lowest sequence number holds the lock, and non-holders watch only their immediate predecessor to avoid the herd effect. Ephemeral nodes provide automatic crash cleanup via session timeout. The recipe supports both exclusive write locks and shared read locks (reader-writer pattern).

### [KDC-003] etcd Concurrency Package — Lease-Based Distributed Mutex with Revision Ordering

**Source Type**: grounded-external

- **Acquisition Event**: ACQ-003
- **Source Anchor**: `tasks/20260401-fix-val-07/acquire/raw-sources/src-005-etcd-concurrency-package.md`
- **Verification Status**: passed
- **Original Excerpt**: > "Purpose: Implements concurrency primitives on top of etcd (distributed locks, elections, STM). Mutex implements the sync.Locker interface backed by etcd."
- **Derived Conclusion**: etcd provides first-class distributed lock primitives through its concurrency package. The Mutex type is backed by a Session (which wraps an etcd lease with automatic keep-alive), and lock fairness is achieved through revision-based ordering. The API implements Go's sync.Locker interface with context-aware cancellation. TryLock provides non-blocking acquisition. Session expiry automatically releases locks, preventing deadlocks from failed clients.
