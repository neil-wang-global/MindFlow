# Mechanism Notes

## 1. Soul 机制

`Soul` 的关键作用不是输出结果，而是约束系统如何定义好结果、偏好何种方法、拒绝什么以及学习什么。它是系统的长期治理源。

## 2. Inference 机制

`Inference` 不应被写成“自由推理”，而应写成“受约束推衍”：

1. 上游工件触发
2. 加载 `Soul`
3. 在知识不完整条件下形成临时判断
4. 结果默认是候选性的，而不是直接固化为能力

## 3. Learning / Capacity 机制

`Learning` 的目标不是堆文档，而是将审核后的知识转化为稳定可调用的 `Capacity`。这使知识管理与执行能力真正打通。

## 4. 执行协议机制

`Execution -> Plan -> Execution Control` 是工程亮点。`Execution` 负责把分析结果形式化为 `Plan`，而 `Execution Control` 则作为 `Execution` 内部的运行控制部分负责推进步骤、管理并行与汇合，结果通过显式文件交接完成。

## 5. 组织场景机制

组织的方法论原则、认知偏好和发展目标进入 `Soul` 之后，系统的学习方向、自主判断和长期演进将不再是无约束的，而是受目标牵引的可控演进。
